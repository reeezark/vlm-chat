# VLM 图文问答助手改进路线执行方案

> 本文档用于指导开发团队将当前项目从“可演示的本地 VLM Chat 应用”逐步升级为“可多人使用、可生产部署、可持续迭代的成熟多模态助手平台”。

## 1. 背景与目标

当前项目已具备以下能力：

- Gradio Web UI
- 多 Provider 模型接入
- 流式输出
- 多图输入
- PDF 文本解析
- SQLite 会话持久化
- 联网搜索工具调用
- 消息重新生成 / 停止 / 复制
- Docker 容器化
- 基础安全限流与生产鉴权保护
- 本地质量门禁脚本

与成熟商业项目相比，主要差距集中在：

- 用户体系与权限隔离不足
- 知识库 / RAG 能力缺失
- 审计、监控、备份恢复能力不足
- 文件安全和数据安全治理不足
- 长上下文、性能缓存、多实例部署能力不足
- 测试、CI/CD、文档和架构分层仍需增强

本路线的目标是按优先级逐步补齐生产化、产品化和工程化能力。

## 2. 改进原则

- **先底座后体验**：优先建设安全、测试、监控、备份等生产底座，再做高级产品能力。
- **先低风险后重构**：先补可独立落地的能力，再拆分架构，避免一次性大重构。
- **每阶段可验收**：每个阶段必须有明确交付物、测试方式和验收标准。
- **配置化优先**：涉及安全、模型、限流、文件大小、存储后端的能力优先通过配置控制。
- **兼容当前使用方式**：保留本地开发和 Docker Compose 的轻量部署方式。

## 4. 阶段 0：建立基线

### 4.1 目标

为后续迭代建立可回滚、可验证、可对比的稳定基线。

### 4.2 任务清单

| 任务 | 说明 | 负责人建议 | 优先级 |
|---|---|---|---|
| 标记稳定版本 | 基于当前通过测试的版本打 tag，例如 `v0.1-baseline` | Tech Lead | P0 |
| 固定运行环境 | 明确 Python 版本、Gradio 主版本、依赖范围 | Backend | P0 |
| 保存评测结果 | 保存当前测试覆盖率、功能截图、评测报告 | QA | P0 |
| 建立改进任务看板 | 将本文档任务拆为 issue / ticket | PM / Tech Lead | P0 |

### 4.3 验收标准

- 当前代码可通过 `scripts/ci_check.sh`
- 当前版本有明确 Git tag
- README 能指导新成员完成本地启动
- 后续任务均可追踪到 issue / ticket

### 4.4 风险与应对

- **风险**：后续改动无法判断是否引入回归。
- **应对**：所有后续阶段都以阶段 0 基线为对照，保留测试报告和评测样例。

## 5. 阶段 1：生产安全与质量底座

### 5.1 目标

让项目具备最低生产部署安全性和基本工程质量保障。

### 5.2 任务清单

| 任务 | 问题描述 | 改进步骤 | 验收标准 | 优先级 |
|---|---|---|---|---|
| 接入远程 CI | 当前只有本地 `ci_check.sh` | 增加 GitHub Actions / GitLab CI，执行编译、测试、覆盖率、Docker build | 每次 PR 自动执行并阻断失败提交 | P0 |
| 增加 Docker Smoke Test | 当前只验证构建，不验证容器启动 | CI 中启动容器并访问首页或健康接口 | 容器启动成功，首页返回 200 | P0 |
| 引入 Ruff / Format | 代码风格依赖人工 | 增加 `ruff check` 和 `ruff format` | CI 可检测代码风格问题 | P1 |
| 增加类型检查 | 仅有部分类型注解 | 引入 `mypy` 或 `pyright`，初期宽松配置 | 关键模块无明显类型错误 | P1 |
| 加强生产鉴权策略 | 已有 Basic Auth，但无用户体系 | 保持生产必配鉴权，同时规划用户表 | 生产外网部署不可无鉴权启动 | P0 |
| 完善限流策略 | 当前按浏览器会话限流 | 增加 IP / 用户 / Provider 维度限流 | 高频请求被限制并有明确错误提示 | P1 |

### 5.3 推荐实现顺序

1. 新增 CI 工作流
2. 将 `scripts/ci_check.sh` 纳入 CI
3. 增加 Docker build 和 smoke test
4. 引入 Ruff，仅在新代码中强制
5. 扩展限流维度

### 5.4 验收标准

- PR 自动执行测试
- 测试失败不能合入
- Docker 镜像能在 CI 中构建
- 生产对外监听时未配置鉴权会拒绝启动

### 5.5 风险与应对

- **风险**：一次性引入 lint 会暴露大量历史问题。
- **应对**：初期只对新增代码强制，历史问题分批修复。
- **风险**：CI 环境无真实 API Key。
- **应对**：CI 中使用 mock 测试和启动 smoke test，不调用真实模型。

## 6. 阶段 2：可观测、审计与备份恢复

### 6.1 目标

让系统具备生产运维所需的状态感知、问题追踪和数据恢复能力。

### 6.2 任务清单

| 任务 | 问题描述 | 改进步骤 | 验收标准 | 优先级 |
|---|---|---|---|---|
| 健康检查接口 | 当前无标准 health endpoint | 增加 `/health` 或 Gradio 外置 FastAPI health route | 部署系统可探活 | P0 |
| 结构化日志 | 当前日志偏运行调试 | 增加请求 ID、session ID、provider、耗时、错误分类 | 日志可按请求串联排查 | P0 |
| 审计日志 | 用户上传、删除、工具调用不可追踪 | 新增 `audit_logs` 表 | 关键操作可追踪到时间和资源 | P0 |
| 指标采集 | 无错误率和耗时统计 | 记录请求耗时、首 token 时间、模型失败率、工具调用次数 | 可输出基础指标 | P0 |
| 告警策略 | 线上异常无感知 | 对错误率、API 失败率、磁盘空间设置告警 | 触发阈值时能通知维护者 | P1 |
| 数据备份 | SQLite 和上传文件无备份 | 增加备份脚本，支持 DB 和 uploads 目录备份 | 可恢复到最近一次备份 | P0 |
| 恢复演练 | 没有验证恢复是否可用 | 增加恢复脚本和演练流程 | 在测试环境完成恢复验证 | P1 |

### 6.3 审计日志建议字段

| 字段 | 说明 |
|---|---|
| `id` | 自增 ID |
| `request_id` | 请求 ID |
| `user_id` | 用户 ID，未接用户体系前可用 session ID |
| `action` | 操作类型，如 upload、chat、tool_call、delete_session |
| `resource_type` | 资源类型 |
| `resource_id` | 资源 ID |
| `provider` | 模型 Provider |
| `model` | 模型名 |
| `success` | 是否成功 |
| `error_code` | 错误码 |
| `created_at` | 操作时间 |

### 6.4 验收标准

- 服务可被健康检查探活
- API 错误、模型调用失败、工具调用失败可通过日志定位
- 删除会话、上传文件、调用工具等操作有审计记录
- SQLite 和上传文件可备份并可恢复

### 6.5 风险与应对

- **风险**：日志记录过多导致隐私泄露。
- **应对**：日志只记录元信息，不记录完整用户内容和文件内容。
- **风险**：备份文件包含敏感信息。
- **应对**：备份目录限制权限，生产环境开启加密存储。

1. 错误提示体系

3. 会话重命名和搜索

### 7.4 验收标准

- 用户能搜索并定位历史会话

- **应对**：短期在 Gradio 内实现关键功能；长期评估迁移到前后端分离。

## 8. 阶段 4：RAG 知识库与文件能力升级

### 8.1 目标

从“临时上传文件问答”升级为“可管理、可检索、可追溯的知识库问答”。

### 8.2 任务清单

| 任务 | 当前差距 | 改进步骤 | 验收标准 | 优先级 |
|---|---|---|---|---|
| 文档库管理 | 仅单次上传 PDF | 新增文档集合、文件列表、删除、重建索引 | 可创建知识库并上传多个文件 | P0 |
| 文档解析增强 | 仅 PDF 文本提取 | 支持 OCR、Word、Excel、CSV、Markdown | 常见文件格式可解析 | P1 |
| 文本切片 | 缺失 | 按标题、段落、token 切分 | 切片可存储和检索 | P0 |
| Embedding | 缺失 | 接入 embedding 模型 | 文档可向量化 | P0 |
| 向量检索 | 缺失 | SQLite 向量扩展 / FAISS / Chroma | 问题可召回相关片段 | P0 |
| 引用来源 | 缺失 | 回答展示引用文档、页码、片段 | 答案可追溯 | P1 |
| RAG 评测 | 缺失 | 构造命中率、引用准确率评测 | 可比较 RAG 效果 | P1 |

### 8.3 技术选型建议

| 组件 | 轻量方案 | 生产方案 |
|---|---|---|
| 文档存储 | 本地文件 + SQLite 元数据 | 对象存储 + Postgres |
| 向量库 | FAISS / Chroma | Milvus / pgvector / Elasticsearch |
| Embedding | DashScope embedding / OpenAI embedding | 可配置多 Provider |
| OCR | PaddleOCR / 云 OCR | 独立 OCR 服务 |

### 8.4 验收标准

- 用户可创建知识库
- 用户可上传多个文件并建立索引
- 提问时能检索相关片段
- 回答中能展示引用来源
- RAG 查询准确率有评测报告

### 8.5 风险与应对

- **风险**：RAG 引入较多新组件，复杂度上升。
- **应对**：先做 SQLite + FAISS/Chroma 的轻量版本，再抽象为可替换后端。
- **风险**：OCR 和文档解析质量不稳定。
- **应对**：保留解析日志和失败重试，允许人工查看解析文本。

## 9. 阶段 5：架构拆分与多实例部署

### 9.1 目标

降低 `app.py` 复杂度，建立可测试、可替换、可多实例部署的架构。

### 9.2 目标架构

```text
src/
  ui/
    gradio_app.py
    theme.py
  services/
    chat_service.py
    file_service.py
    rag_service.py
    audit_service.py
  providers/
    base.py
    dashscope.py
    openai_provider.py
    openrouter.py
    ollama.py
  repositories/
    session_repository.py
    message_repository.py
    audit_repository.py
    document_repository.py
  security/
    auth.py
    rate_limit.py
    validators.py
  observability/
    metrics.py
    request_context.py
```

### 9.3 任务清单

| 任务 | 当前问题 | 改进方向 | 验收标准 | 优先级 |
|---|---|---|---|---|
| 抽离样式 | CSS/JS 内联在 `app.py` | 移到 `src/ui/theme.py` 或静态 CSS | `app.py` 行数下降 | P1 |
| 抽离 ChatService | 业务流程在 UI 类中 | `process_query` / `regenerate` 下沉服务层 | 服务层可单测 | P1 |
| 抽离 Repository | SQLite 操作集中在 ChatManager | 存储接口化 | 可替换 Postgres | P1 |
| 拆分 Provider | `api_client.py` 逐渐膨胀 | 每个 Provider 独立类 | 新增模型无需改主流程 | P1 |
| 支持 Postgres | SQLite 不适合多实例 | 新增 Postgres Repository | 多实例共享会话 | P1 |
| 支持对象存储 | 上传文件本地化 | 接入 OSS/S3/TOS | 多实例共享文件 | P1 |
| Redis 限流 / 状态 | 进程内限流无法跨实例 | Redis 存储限流桶和短状态 | 多实例限流一致 | P2 |

### 9.4 拆分顺序

1. 抽离 theme 和 CSS
2. 抽离 security / rate_limit
3. 抽离 chat_service
4. 抽离 repositories
5. 抽离 providers
6. 增加 Postgres / 对象存储实现

### 9.5 验收标准

- `app.py` 只负责 UI 组装和事件绑定
- 核心业务逻辑可脱离 Gradio 单测
- SQLite 和 Postgres 可通过配置切换
- 本地文件和对象存储可通过配置切换

### 9.6 风险与应对

- **风险**：大重构导致功能回归。
- **应对**：按职责逐步迁移，每一步都跑 `scripts/ci_check.sh`。
- **风险**：抽象过度。
- **应对**：只为明确要替换的模块建立接口，不为短期不会变化的部分过度设计。

### 16.4 本次变更说明

- 保留当前 Gradio 使用方式；当 `GRADIO_SHARE=true` 时继续使用 `demo.launch()`，当 `GRADIO_SHARE=false` 时使用 FastAPI + `gr.mount_gradio_app()` 提供 `/health`。
- 审计日志只记录元信息，不记录完整用户问题、模型回答或文件内容，降低隐私泄露风险。
- 备份恢复默认需要 `--force` 才会覆盖当前数据，避免误恢复。
- CI 中使用 dummy API Key，仅验证服务能启动和 health endpoint 可用，不调用真实模型完成问答。
- RAG 当前采用 SQLite + 本地哈希 embedding，适合作为 MVP 和后续向量库替换前的最小可用链路，不等同于成熟云端 embedding 语义检索。
- 用户归属当前基于 `DEFAULT_USER_ID` / `DEFAULT_USERNAME` 默认用户上下文，目的是先建立数据模型与过滤边界；真实多用户仍需接入登录态、Session/JWT 和 RBAC。
- 前端自定义模型 API 的 API Key 仅进入运行时 `VLMAPIClient`，不写入 SQLite 和审计日志；生产环境仍建议结合登录鉴权和用户级密钥管理。
- 自定义模型 API 已从全局客户端切换为会话级客户端池，浏览器会话删除时同步清理对应运行时客户端和限流桶。
- React 前端采用渐进式迁移策略：`FRONTEND_MODE=react` 时根路径提供 React 应用，Gradio 保留在 `/gradio`；默认仍可用 Gradio 模式以便回滚。
- RAG 入库现在会清洗 PDF 抽取文本中的非法 Unicode 字符，并以事务方式写入文档和切片，避免出现“文档已入库但无切片可检索”的状态。
- OpenAI 兼容 Provider 现在支持通过 `MODEL_NAME`、`OPENAI_MODEL_NAME` 或 `OPENAI_MODELS` 配置默认模型和前端模型列表；`.env` 已切换为 `DEFAULT_PROVIDER=openai`。
- React 前端的自定义 API 地址和 API Key 可保存到浏览器 `localStorage`，便于个人设备重复访问时自动回填；该 Key 为浏览器本地明文存储。
- 模型配置已新增统一解析与校验层，React 前端可在发送消息前独立执行“校验配置”和“测试连接”，配置摘要不会返回 API Key 原文。
