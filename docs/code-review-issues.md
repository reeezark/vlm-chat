## 一、安全性问题

### SEC-01 监听 0.0.0.0 但无任何鉴权

- **优先级**：P0
- **位置**：`src/config.py:35`，`app.py:378-384`
- **问题描述**：默认 `GRADIO_SERVER_NAME = "0.0.0.0"`，服务对外暴露在所有网卡上，且 `demo.launch` 未配置任何 `auth`，任何能访问该端口的人均可使用应用并消耗 API 额度。
- **出现场景**：在共享网络/服务器上启动应用即生效。
- **相关上下文**：API Key 通过环境变量加载（`src/config.py:16`），但接口本身无访问控制。
- **建议**：本地默认绑定 `127.0.0.1`；如需对外，通过环境变量控制并在 `launch` 中增加 `auth=`。

### SEC-02 图片校验仅查扩展名，未校验真实内容

- **优先级**：P0
- **位置**：`src/image_processor.py:25-48`，`src/api_client.py:52-60`
- **问题描述**：`validate_format` 仅用 `Path.suffix` 判断格式；MIME 类型也靠扩展名猜测。伪装扩展名的恶意或损坏文件可绕过校验，被直接 base64 编码发送给 API。Pillow 已是项目依赖却未用于内容校验。
- **出现场景**：用户上传将 `.exe`/损坏文件重命名为 `.jpg` 的文件。
- **相关上下文**：`validate_image` 仅串联格式与大小校验，无像素级解析。
- **建议**：使用 `PIL.Image.open().verify()` 校验真实图像，并据真实格式推断 MIME。

## 二、正确性与功能问题

### FUN-01 全局单例会话状态导致多用户串话

- **优先级**：P0
- **位置**：`app.py:203-213`，`src/chat_manager.py:28-29`
- **问题描述**：`VLMChatAssistant` 仅实例化一次，`ChatManager.current_session_id` 与 `current_image` 为进程级全局字段。Gradio 默认多用户共享后端，一个用户切换/新建会话会直接影响其他在线用户看到的内容。
- **出现场景**：两个及以上浏览器/用户同时访问应用。
- **相关上下文**：`process_query`、`switch_session`、`create_new_session` 等均依赖 `self.chat_manager.current_session_id`。
- **建议**：使用 `gr.State` 在每个浏览器会话保存 `session_id`，将 `current_session_id` 从 `ChatManager` 移除，所有方法显式传入 `session_id`。

### FUN-02 图片路径持久化为本地临时路径，跨会话/跨机失效

- **优先级**：P0
- **位置**：`src/chat_manager.py:136-161`、`243-262`
- **问题描述**：会话中存储的是 Gradio 临时路径（如 `C:\Users\...\Temp\gradio\...webp`）。应用重启或更换机器后 `os.path.exists` 为假，历史图片彻底丢失。`ImageProcessor.save_temp_image`/`cleanup_temp_image` 已定义但从未被调用。
- **出现场景**：上传图片→重启服务→查看历史，图片无法显示。
- **相关上下文**：`load_all_sessions` 加载时依赖原图存在以重建缩略图。
- **建议**：上传后复制图片到项目内持久目录（如 `data/uploads/`），持久化相对路径。

### FUN-03 API 失败时会话状态不一致

- **优先级**：P1
- **位置**：`app.py:251-258`
- **问题描述**：先 `add_message(user)` 落库，再调用 API；若 API 抛错，用户消息已持久化但无对应助手回复；`except` 分支又向 `chatbot_history` 追加一条游离消息，与持久化历史不同步。
- **出现场景**：网络异常或 API 报错时发送消息。
- **相关上下文**：`chat_manager.add_message` 每次都会写盘。
- **建议**：成功后再统一落库；或失败时回滚已写入的 user 消息，保证 UI 与存储一致。

## 三、性能问题

### PERF-01 会话 JSON 持久化了超大 base64 缩略图

- **优先级**：P0
- **位置**：`src/chat_manager.py:124-134`、`223-241`
- **问题描述**：`save_session` 直接写入含 `image_thumbnail`（完整图片 base64）的 `history`。实测单个 session 文件达 26 万+ 字符。所谓“缩略图”实为原图全量编码，未做任何压缩；而 `load_all_sessions` 已能在加载时从原图重建缩略图，写入磁盘属冗余。
- **出现场景**：含图片的会话每次新增消息都会重写整个大 JSON。
- **相关上下文**：示例文件 `data/sessions/440e65...json` 体积异常。
- **建议**：保存时剔除 `image_thumbnail` 字段；用 Pillow `thumbnail((200,200))` 生成真正的小缩略图。

### PERF-02 每次请求对历史图片重复 base64 编码

- **优先级**：P1
- **位置**：`src/api_client.py:99-118`、`133-141`
- **问题描述**：`build_messages` 对历史中每张图片都重新读盘并 base64 编码，多轮对话开销随轮数线性增长。
- **出现场景**：多轮带图对话。
- **相关上下文**：`MAX_HISTORY_LENGTH = 10`，历史可累积多张图片。
- **建议**：缓存编码结果，或历史仅保留必要图片；也可改为传可访问 URL。

### PERF-03 无流式输出，响应体验差

- **优先级**：P1
- **位置**：`app.py:251-255`，`src/api_client.py:182-192`
- **问题描述**：使用阻塞式 `chat.completions.create`，用户须等待完整生成后才能看到回复，首字延迟高。
- **出现场景**：每次问答，尤其是长回答。
- **相关上下文**：`MAX_TOKENS = 512`。
- **建议**：改用 `stream=True` 配合 Gradio generator（`yield`）实现流式输出。

## 四、健壮性问题

### ROB-01 重试策略固定 sleep，无指数退避

- **优先级**：P1
- **位置**：`src/api_client.py:179-200`
- **问题描述**：固定 `time.sleep(2)` 重试，且对所有异常（含 401 鉴权等不可恢复错误）一律重试，浪费时间且无意义。
- **出现场景**：API 鉴权失败或参数错误时。
- **相关上下文**：`max_retries = 3`。
- **建议**：采用指数退避，并仅对网络/限流类错误重试。

### ROB-02 过期会话与临时图片清理不彻底

- **优先级**：P1
- **位置**：`app.py:211`，`src/chat_manager.py:271-278`
- **问题描述**：`cleanup_expired_sessions` 仅在 `__init__` 启动时执行一次，长时间运行的服务不会再清理；临时图片目录无任何清理机制。
- **出现场景**：服务长期运行后磁盘累积过期会话与临时图。
- **相关上下文**：`SESSION_TIMEOUT = 3600`。
- **建议**：定时清理过期会话与临时图片。

## 五、代码质量与可维护性

### QUA-01 重复的 MIME 推断逻辑

- **优先级**：P2
- **位置**：`src/api_client.py:52-60`，`src/chat_manager.py:129-131`
- **问题描述**：扩展名→MIME 的判断逻辑在两处各写一遍，存在重复。
- **建议**：抽取到 `image_processor` 统一维护。

### QUA-02 巨型内联 CSS/JS 字符串

- **优先级**：P2
- **位置**：`app.py:35-200`
- **问题描述**：约 170 行样式硬编码于 Python 字符串中，难以维护与复用。
- **建议**：抽成独立 `.css` 文件，使用 `gr.Blocks(css=...)` 加载。

### QUA-03 defaultdict(dict) 使用不当

- **优先级**：P2
- **位置**：`src/chat_manager.py:28`
- **问题描述**：`self.sessions = defaultdict(dict)`，对未知 key 的读取可能意外创建空会话；当前靠 `.get` 规避，仍为隐患。
- **建议**：改用普通 `dict`。

### QUA-04 日志模块强制全局禁用

- **优先级**：P2
- **位置**：`src/logger.py:11-19`
- **问题描述**：直接将 httpx/openai 等 logger `disabled=True`，调试时难以排查底层请求问题。
- **建议**：通过日志级别/环境变量控制输出，而非硬禁用。

## 六、开发体验与文档

### DEV-01 requirements.txt 缺失 jieba

- **优先级**：P3
- **位置**：`evaluation/evaluator.py:5`，`requirements.txt`
- **问题描述**：评测脚本依赖 `jieba`，但 `requirements.txt` 未列出，运行评测会直接 ImportError。
- **建议**：在 `requirements.txt` 中补充 `jieba`。

### DEV-02 依赖版本与代码实际使用不符

- **优先级**：P3
- **位置**：`requirements.txt:4`，`app.py`
- **问题描述**：requirements 写 `gradio>=4.0.0`，但代码使用 `MultimodalTextbox`、messages 字典格式及注释“Gradio 6.x”，存在 API 不兼容风险。
- **建议**：锁定主版本（如明确 6.x 或 `>=4,<5`），与实际代码一致。

### DEV-03 README 与实际结构不符

- **优先级**：P3
- **位置**：`README.md:56-85`
- **问题描述**：README 描述了 `static/examples/`、`data/temp_images/` 等不存在的目录，且结构图缺少 `logger.py`、`tests/`、`scripts/`、`docs/`。
- **建议**：同步更新项目结构说明。

### DEV-04 配置项硬编码、不可调

- **优先级**：P3
- **位置**：`src/config.py:26-32`，`src/chat_manager.py:20`
- **问题描述**：`TEMPERATURE/MAX_TOKENS/MAX_HISTORY_LENGTH` 等写死，无法经环境变量/UI 调整；`HISTORY_STORAGE_DIR` 硬编码在 `chat_manager` 内而非走 config。
- **建议**：关键参数支持环境变量覆盖，存储目录统一到 config。

### DEV-05 测试覆盖缺口

- **优先级**：P3
- **位置**：`tests/`
- **问题描述**：仅覆盖 `chat_manager` 等，缺少 `process_query`、多用户隔离、API 失败回滚等关键路径测试；`test_save_and_load` 依赖真实目录加载，隔离性弱。
- **建议**：补充关键路径与隔离性测试。
